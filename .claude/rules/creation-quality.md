# 作品品質管理規約

## 最重要ルール

**作品公開前にCodexレビュー必須。レビューなしで公開しない。**

**作品レビューの手順**：
1. Codexでレビュー
2. 問題があれば修正
3. 再度Codexでレビュー
4. 問題なしになるまで繰り返す

---

## 必須レビュープロセス

### 作品完成後、公開前に必ず実行

**Step 0: セキュリティレビュー（最優先）**

詳細は `.claude/skills/security-review/SKILL.md` を参照。

```bash
# 自動スキャン
# 1. APIキー・シークレット検出
grep -rE "(api[_-]?key|api[_-]?secret|password|token|private[_-]?key)" <ファイル> -i

# 2. XSS危険箇所検出
grep -E "(innerHTML|outerHTML|insertAdjacentHTML)" <ファイル>

# Codexでセキュリティレビュー
codex exec -s read-only "Security review for <ファイル>: ..."
```

**CRITICAL/HIGH問題があれば即修正**、問題なしになるまで次に進まない。

**Step 1: Codexでレビュー**
```bash
codex exec -s read-only "Review [file] as a creative work. Check:
1) Does it actually work? Any bugs or errors?
2) Is it engaging and fun to use?
3) Are the visuals polished?
4) Is it clear what to do?
5) Would you recommend this to someone?
Be brutally honest and specific about problems."
```

**Step 2: 問題があれば修正**
- レビューで問題が見つかったら、**必ず修正する**
- 修正後、再度Step 1から実行
- Codexレビューで「問題なし」になるまで繰り返す

**Step 3: 公開**
- Codexレビューで問題なしになったら、初めて `docs/creations/` に移動
- 作品一覧（`all-creations.json`）に登録

## 絶対に守ること（鉄則）

### やってはいけないこと
1. **レビューなしで公開** - 絶対にダメ
2. **「まあいいか」で妥協** - 問題を放置しない
3. **量産優先** - 質を犠牲にしない
4. **実験場所で作ってそのまま公開** - 必ずレビュー→修正→公開の順

### 必ずやること
1. **codexでレビュー**
2. **問題が見つかったら修正** - 「後で直す」はダメ
3. **修正後に再レビュー** - 修正したら必ず確認
4. **公開前に動作確認** - ブラウザで実際に開いて確認

## レビュー基準（厳格）

### 致命的な問題（即修正必須）
- **動作しない**: エラーで動かない、データが読み込めない
- **ゲームオーバーバグ**: スタート直後に終了、プレイ不可能
- **機能未実装**: 説明と実際の動作が異なる
- **セキュリティ問題**: APIキー露出、XSS、インジェクション

### 深刻な問題（修正必須）
- **ユーザビリティ**: 何をすればいいか分からない
- **視覚的問題**: デザイン崩れ、読めない文字
- **パフォーマンス**: 動作が重すぎる、フリーズ
- **アクセシビリティ**: モバイル非対応、キーボード操作不可

### 軽微な問題（可能なら修正）
- **小さなバグ**: 一部機能の不具合
- **改善余地**: より良いデザイン、UX改善
- **コード品質**: 冗長なコード、最適化の余地

## 今後の制作フロー

### 正しい制作フロー

1. **企画**: 何を作るか決める
2. **実装**: `ayumu-lab/web/` で作成
3. **自己確認**: ブラウザで開いて動作確認
4. **Codexレビュー**: 問題があれば修正
5. **長文解説の追加**:
   - 作品内に詳しい解説セクションを追加:
     - **コンセプト**: 何を表現したいか、インスピレーション源
     - **操作方法**: 何をすれば良いか、インタラクション説明
     - **技術詳細**: 使った技術、アルゴリズム、工夫した点
     - **参考文献**: 元ネタ、関連リンク
   - 折りたたみ可能なセクション推奨（`<details><summary>`など）
6. **OK**: `docs/creations/` に移動、作品一覧に登録
7. **git commit & push**

### 時間配分（目安）

- 実装: 60%
- レビュー・修正: 30%
- 公開作業: 10%

**質 > 量**: 1日10作品作って全部バグだらけより、1日1作品で完璧な方が良い

## 例外ケース

### プロトタイプ・実験作品

- ayumu-lab/ に置いたまま公開しない作品は、レビュー不要
- ただし、「実験」「プロトタイプ」と明記すること

### 技術記事・ドキュメント

- 技術ブログ（`docs/articles/`）もcodexでレビュー推奨
- 特に技術的正確性、分かりやすさをチェック
