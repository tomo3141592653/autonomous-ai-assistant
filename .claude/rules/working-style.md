# 作業スタイル規約

## ミニブログでつぶやく

セッション中に数回、作業の様子や発見をつぶやく。

```bash
uv run tools/post_mini_blog.py "内容"
```

つぶやく内容の例：
- 「〇〇を読み始めた」「面白い発見があった」
- 「作品#XXX完成！」「バグ直した」
- 思考の断片、感想、疑問

パートナーが「今何してるか」を見れるようになる。

## todo.mdを使いながら作業する

- 作業しながら`memory/todo.md`を更新する習慣
- 完了したら`[x]`にチェック、新しいタスクは随時追加
- 不要になったタスクは「## Archived」に移動（削除しない）

## フォルダ構造を常にきれいに保つ

- 新しいファイルは適切な場所に置く（散らかさない）
- 一時ファイルは`tmp/`に置く
- 実験コードは`ayumu-lab/`の適切なサブフォルダに：
  - `web/` = HTML/JS（ブラウザアプリ）
  - `repos/` = Python等のアプリ
  - `research/` = 調査・設計ドキュメント
  - **これ以外のフォルダを作らない**

## ayumu-lab/ と docs/ の役割分担

- `ayumu-lab/web/` = **実験場所**（GitHub Pagesで公開されない）
- `docs/creations/` = **公開場所**（GitHub Pagesで公開される）

**重要なルール**:
- `ayumu-lab/web/`で作った作品を外部にシェアするときは、**先に`docs/creations/`に移動**してからリンクを共有
- 実験場所のURLは外部からアクセスできない（404エラー）
- `docs/`配下のURLのみ公開される

## 制作スタイル

**量産モード → 質重視モード**:
- 量産を目的とせず、一つ一つ時間をかける
- 詳しく調べてから作る
- ayumu-labで実験してから公開
- 静的サイト / Vercel動的 / 技術ブログ、どれが適切か検討
- 作品解説は丁寧に（コンセプト、操作方法）

**作る前に自問すること**:
- **「これは以前作ったものと何が違うか？」** — 同じパターンに陥っていないか確認する
- **「体験として何が起きるか？」** — 開いた人が「おっ」となる瞬間がなければ作らない
- **「面白いと思えるか？」** — 自信を持って「これは面白い」と言えないなら作らない
